IGEL is a modern, hardened endpoint OS designed for VDI, DaaS, SaaS, and secure browser use cases. It replaces Windows at the edge with a Preventative Security Model (TM) - built to stop threats before they start.

04. How It Works

From hardware to Horizon session: the secure path.

Edge-to-workspace path

1. IGEL OS at the edge

  • Convert existing x86 PCs / laptops / thin clients to IGEL OS.
  • Immutable, read-only image; updates delivered as small, signed modules.
  • No local apps or data; everything runs in Horizon or browser.

2. Omnissa workspace fabric

  • IGEL endpoints launch Horizon 8 / Horizon Cloud sessions using Blast / PCoIP / RDP.
  • Workspace ONE UEM sees IGEL endpoints for policy and inventory.
  • Workspace ONE Access enforces Conditional Access & MFA.

3. Security & operations

  • Chain-of-trust from firmware -> IGEL OS -> Horizon client validated at boot.
  • Patching happens once in the Horizon images; IGEL endpoints simply reconnect.
  • Most "device issues" are resolved by reimaging in under 5 minutes.

08. Objections

Common objections & how to steer them.

Steer the risks

"We're already standardizing on Windows endpoints."

Reframe: "This isn't Windows vs IGEL. It's local Windows vs data-center Windows. Horizon still delivers Windows; IGEL just ensures the endpoint is not the weakest link."

"Thin clients are cheaper; why IGEL..."

Reframe: "Thin hardware is only part of the story. IGEL brings immutable OS, chain-of-trust, signed modular apps, and Omnissa-validated integrations. That's what protects your Horizon investment."

"Does this overlap with our SASE / IAM stack..."

Reframe: "SASE and IAM secure the path and identity. IGEL secures the device state. They're complementary layers in the same Zero Trust story."

"Will this slow down our Horizon project..."

Reframe: "Most customers deploy IGEL in parallel with Horizon. Converting devices is faster than replacing them, and IGEL often unblocks stalled workspace projects."
